Architecture Decision Record v1.0

Why it is built
this way.

Governance records the rules; this records the reasoning. Each entry states a decision, the alternatives that were rejected, and — deliberately — the trade-off it accepts. The most expensive thing to reconstruct in any long-lived asset is why, so it is written down.

ADR-0001

Static site with client-side checks, no backend

Accepted
Context

The core tool grades a domain's email authentication. That could run on a server or in the browser.

Decision

Ship a fully static site; run domain checks in the browser over DNS-over-HTTPS. No server, no accounts, no storage.

Alternatives rejected
  • A server API doing the DNS lookups. Adds hosting, rate-limiting, logging and a privacy surface; a domain's grade and the visitor are never ours to hold if there is no server.
  • Server-side rendering. Unnecessary for content that is deterministic and cache-friendly; it would trade the privacy guarantee for marginal speed.
Trade-off accepted

Checks are bounded to what public DNS over HTTPS and the browser allow — most notably the DKIM selector heuristic (ADR-0008). There is no place to persist per-user history.

ADR-0002

Git history as the Observatory datastore

Accepted
Context

The Observatory needed to become a dated record, not a per-visit sample — which implies storage of measurements over time.

Decision

Write each weekly measurement as JSON into the repository (latest.json + an immutable history/ archive). Git history is the audit trail.

Alternatives rejected
  • A key-value store (e.g. Cloudflare KV). Introduces an external dependency, credentials and a retention policy to manage, for data that is small, append-mostly and benefits from being openly diffable.
  • An external database. Far heavier than the data warrants, and it would move the record out of the open, versioned artifact the rest of the asset already is.
Trade-off accepted

Granularity is weekly, not real-time; there is no query API; and the repository grows over time (bounded — snapshots are ~2 KB and git-compressed). If sub-weekly or queryable history is ever needed, this decision would be revisited.

ADR-0003

Deterministic build scripts over a framework

Accepted
Context

Pages are generated from data (techniques, patterns, records). Something has to turn data into HTML.

Decision

Use small, dependency-free Node scripts that read data and emit complete HTML, verified by one gate.

Alternatives rejected
  • A static-site generator (Hugo, Astro, Eleventy). Adds a toolchain and dependency tree to maintain and audit, for output we can produce exactly with a few readable scripts.
  • A client framework (Next, SvelteKit). Ships a runtime and build system to render content that is fundamentally static; it works against the privacy and longevity goals.
Trade-off accepted

Componentisation is manual and there is no plugin ecosystem — shared markup is duplicated across builders and kept honest by the gate rather than by a template engine.

ADR-0004

Permanent, structured identifiers

Accepted
Context

Techniques, patterns and decisions are cited by identifier. Citations must keep resolving.

Decision

Assign structured, namespaced identifiers (e.g. SPO-AU-03, EAP-01, ADR-0001) that are permanent: never reused or renumbered; superseded entries are deprecated, not deleted.

Alternatives rejected
  • Human slugs as the primary key. Slugs drift as wording is refined, breaking citations; they are good URLs but poor identifiers.
  • A single global sequence. Loses the pillar/phase structure that makes an id self-describing and sortable.
Trade-off accepted

Identifiers cannot be tidied or renumbered even when the set would look cleaner; gaps and deprecations are permanent. Durability is bought with a little untidiness.

ADR-0005

Pillar pages are the source; model.json is generated

Accepted
Context

The technique corpus exists both as readable pages and as machine-readable model.json. One must be authoritative to avoid drift.

Decision

Treat the built pillar pages as the source of record and derive model.json from them; the gate checks the JSON conforms and stays in sync.

Alternatives rejected
  • JSON as source, pages generated from it. Equally valid, but it makes the human-readable, canonical artifact a by-product of a data file — the opposite of a reference whose pages are the citable object.
Trade-off accepted

The JSON must be regenerated whenever page content changes, and that discipline is enforced by process and the gate rather than being impossible to get wrong.

ADR-0006

One canonical Organization across the knowledge graph

Accepted
Context

Every page emits JSON-LD. Search and knowledge graphs resolve entities by identity.

Decision

Define a single Organization entity and reference it by a shared @id on every page, rather than inlining it.

Alternatives rejected
  • Inline the Organization on each page. Produces many near-duplicate entities that a consumer cannot reliably unify into one publisher.
Trade-off accepted

Every page must carry the identical @id, and a publisher change means touching all of them — a constraint the gate enforces so it cannot silently diverge.

ADR-0007

One deterministic gate as the definition of valid

Accepted
Context

The asset has many invariants: links, structured data, schema conformance, cross-artifact sync.

Decision

Consolidate every invariant into one script (scripts/verify.js) that runs in CI and blocks merge on any failure.

Alternatives rejected
  • A collection of linters and ad-hoc checks. Spreads the definition of correct across tools and configs; nothing owns the cross-artifact invariants that matter most here.
  • Manual review. Not reproducible and does not scale to hundreds of links and several machine-readable datasets.
Trade-off accepted

The gate is bespoke and must itself be maintained and trusted; a bug in it is a single point of failure — mitigated by keeping it small, and by negative-testing new invariants when they are added.

ADR-0008

Two grading scores instead of one

Accepted
Context

DKIM cannot be read exactly from DNS without knowing the selector; SPF, DMARC and MX can.

Decision

Keep two scores: the full four-control check (with a bounded DKIM selector heuristic) for a single domain, and a DKIM-agnostic three-control index, rescaled to 100, for the Observatory at scale.

Alternatives rejected
  • A single unified score everywhere. Either drags the DKIM heuristic into the at-scale index (introducing false negatives across many domains) or drops DKIM from the single-domain grade (losing a real control).
Trade-off accepted

Two numbers exist for the same domain and must be reconciled; a perfect-but-no-DKIM domain reads A+ in the index and B in the full check. The reconciliation is stated explicitly rather than hidden.

ADR-0009

Defensive-only scope as an absolute boundary

Accepted
Context

The intelligence model describes how email abuse works. Depth is valuable; operational detail is dangerous.

Decision

Keep every entry at the level of classification and defence — what it is, how it is observed, how it is countered — with no payloads, kits or step-by-step methods, at every version, by policy.

Alternatives rejected
  • Include operational/offensive detail for completeness. Would convert a defensive reference into an abuse enabler; it is out of scope regardless of demand.
Trade-off accepted

Readers seeking offensive tradecraft will not find it here; that audience is deliberately unserved.

ADR-0010

Security headers via meta-CSP plus edge rules

Accepted
Context

The site is hosted on GitHub Pages (which cannot set response headers) and fronted by Cloudflare (which can).

Decision

Ship a meta http-equiv Content-Security-Policy in every page so protection holds regardless of host, and mirror the header set at the Cloudflare edge.

Alternatives rejected
  • Rely on a _headers file. Inert on GitHub Pages — it would give a false sense of protection with none applied.
  • Move to a host that sets headers. Trades the simplicity and durability of static hosting for a capability a meta-CSP already provides.
Trade-off accepted

The policy lives in two places (page meta and edge rules) that must be kept aligned; the meta-CSP is the guaranteed floor, the edge rules the enhancement.

How this record works

Recorded, versioned, and kept honest.

A decision is added when a choice is load-bearing — when reversing it later would be costly, or when a future maintainer would otherwise have to reverse-engineer the reasoning. Trivial choices are left out on purpose.

Decisions are not deleted. If one is reversed, a new record supersedes it and the original stays on file, exactly as identifiers are deprecated rather than removed under governance. The record is published as decisions.json (CC BY 4.0) and kept in sync with this page by the quality gate.
